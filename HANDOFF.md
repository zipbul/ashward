# crucible — 핸드오프

작성일 2026-07-03 · 상태: 방향만 확정, 설계 미완(논의 필요) · 위치 `projects/zipbul/crucible/`

## 확정된 것 (사용자 결정)

- **crucible = npm 설치형 소비자 패키지.** 테스트 서포트 라이브러리. CLI 없음.
- 소비자는 패키지를 설치하고, **패키지에 내장된 규칙**을 선언만 하면 자동으로 실제 요청을 보내 국제표준 준수를 테스트한다.
- 규칙은 crucible 패키지 자체에 들어있다. **zipbul의 STANDARDS.md를 읽거나 빌드하지 않는다** (이건 이전 설계자의 오해였음 — 폐기).
- 타깃은 그냥 **소비자**. 앱이든 서버든 구분 안 한다.
- 나중에 **보안 테스트 기능**도 같은 방식으로 추가될 예정(앱/서버 무관).

## 폐기된 오해 (반복 금지)

- ❌ "STANDARDS.md → rules.json 빌드" — 사용자가 지시한 적 없음. crucible은 설치형이라 규칙을 자체 보유.
- ❌ "corpus를 zipbul에서 crucible로 옮긴다" — 잘못된 전제.
- ❌ 타깃을 앱저자 vs HTTP구현자로 쪼개는 논쟁 — 사용자가 명시 거부. 소비자는 소비자.

## 기존 산출물

- `crucible/PLAN.md` — v2 계획서. **위 오해가 섞여 있으니 그대로 신뢰하지 말 것.** 재작성 대상.
- 삼자 리뷰 2라운드 완료. 유효하게 남은 순수 기술 지적(타깃·corpus 무관):
  - `crucible(test)`가 `test` 하나만 받으면 피검 서버 기동/종료 훅을 못 심음 → 훅 번들 주입 필요.
  - 규칙 test 자동 주입 시 동기 등록 ↔ 비동기 서버 기동 순서 계약 필요.
  - 소비자가 특정 규칙 끄기/예외 선언하는 override API 필요.
  - 규칙팩 버저닝(패키지 업뎃으로 새 규칙 추가 시 소비자 CI가 깨지는 문제).
  - 규칙 필터 API, 규칙 ID 타입 안전성.
  - v1은 과잉 잘라내기: FIN/RST 4-way 구분·다중팩 dedup·바이트단위 픽스처 규약은 불필요/과장.
